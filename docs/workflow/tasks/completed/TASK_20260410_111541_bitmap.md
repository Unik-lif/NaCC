## Planner Route

### Metadata Model

- Use a 2-bit PFN tag model:
  - `00 = NORMAL`
  - `01 = ROOT_L0`
  - `10 = PRIVATE_DATA`
  - `11 = RESERVED`
- Bitmap coverage in this round:
  - satp root / PGD pages that need partial lower-half protection
  - confidential user private data pages reached through user leaf PTEs
- Explicitly out of bitmap scope:
  - secure non-leaf PTP pages in the secure PTP region
  - agent pages
  - monitor metadata pages

### Minimal Mapping Records

- Add a small monitor-side root-record table keyed by root PFN, with only coarse fields such as:
  - `valid`
  - `root_pfn`
  - `cid`
  - `flags`
  - optional `family_root_pfn` if it helps same-CID fork bookkeeping
- Purpose of the root-record table:
  - root-tag lifecycle and retirement
  - coarse debug visibility
  - optional last-CID / last-root cleanup decisions
- Non-purpose:
  - do not turn it into a per-page owner table
  - do not mirror VMAs
  - do not track every mapping edge

### Event To Action Mapping

- Confidential container registration:
  - keep current CID / PID registration path
  - no PFN tagging yet
- New NaCC process/mm creation:
  - tag the root PGD PFN as `ROOT_L0` when `pgd_alloc()` completes for the new NaCC process/mm
  - this is the authoritative first protection point required by this packet
- Invoke / exec / fork-child attach after secure non-leaf validation:
  - do not rely on this stage as the first `ROOT_L0` tag point
  - this stage may validate, refresh, or confirm the already-protected root state
  - refresh / rebuild a root record for that PFN and CID
  - walk the user half and tag present user leaf PFNs as `PRIVATE_DATA`
  - leaf/private-data tagging may still remain later than that early `ROOT_L0` point
- NaCC address-space setup while `ROOT_L0` is already active:
  - direct Linux lower-half root writes should be blocked by the tag-aware QEMU check
  - the blocked attempted lower-half write should be the narrow control event that hands execution back to the monitor path
  - the required L0 update should then proceed through a controlled OpenSBI-owned write path so the NaCC address space is arranged correctly without reopening free Linux writes
  - do not broadly rewrite Linux root/PUD setter functions into explicit root-write request helpers as the main mechanism
- Secure leaf install through `sm_nacc_set_ptes()`:
  - if the new PTE is a valid user leaf to a non-secure-PTP PFN, set `PRIVATE_DATA`
  - this is an acceptable later timing point for leaf/private-data tagging in this packet
- Same-CID fork duplication:
  - child root gets its own `ROOT_L0` tag
  - inherited / reused leaf PFNs stay `PRIVATE_DATA`
  - do not add per-page sharing metadata in this round
- MM teardown:
  - retire the dead root PFN tag / record on a controlled path
  - do not interpret `exit_mmap()` entry by itself as proof that the root page is safe to downgrade to `NORMAL`
  - do not add precise per-page `PRIVATE_DATA` untagging if that requires allocator-wide surgery
- CID / container teardown:
  - if a cheap last-PID notification path is practical in this cut, use it for coarse remaining cleanup
  - otherwise leave explicit TODOs and keep Phase A bounded
