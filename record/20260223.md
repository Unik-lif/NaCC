## NaCC 渐进式测试计划

### 设计原则
根据NaCC系统的架构特点（QEMU PMP模拟 → OpenSBI PTP分配/回收 + store/amoswap模拟 → Linux VM_NACC页表管理 → Agent），测试应该**逐步增加复杂度**，每一层都覆盖新的内核路径和内存管理场景。

### 当前进度
- [x] **Level 0**: `docker run hello-world` — 最基本的容器生命周期（创建、执行、退出）
- [x] **Level 1**: `docker run --rm busybox echo test` — busybox动态链接 + 简单syscall

---

### Level 2: 基本文件系统与I/O操作
**目标**: 触发更多的page fault路径、文件读写相关的mmap/munmap
```bash
# [Done] 2a: 文件系统读取 — 触发文件映射(filemap)相关的page fault
docker run --rm busybox ls /

# 2b: 文件系统写操作 — 触发写时复制(CoW)和overlay FS的写入路径
docker run --rm busybox sh -c "echo hello > /tmp/test.txt && cat /tmp/test.txt"

# 2c: 管道操作 — 触发pipe相关的内存分配
docker run --rm busybox sh -c "echo hello | cat"
```
**关注点**:
- overlay FS的page table操作（upperdir/lowerdir的合并）
- `mmap` 文件映射过程中PTP的分配和管理
- 管道(pipe)缓冲区的内存分配是否正确

---

### Level 3: 多进程 / fork + exec
**目标**: 验证fork时的页表复制（copy_page_range）和COW机制
```bash
# 3a: fork + exec 基础场景
docker run --rm busybox sh -c "echo start && echo end"

# 3b: 多个子进程 — 多次fork的页表管理
docker run --rm busybox sh -c "for i in 1 2 3; do echo proc_\$i; done"

# 3c: 后台进程 — 并发的内存分配和释放
docker run --rm busybox sh -c "sleep 1 & echo foreground && wait"
```
**关注点**:
- `copy_page_range` 中对NaCC PTP的正确处理
- 多个子进程退出时的 `exit_mmap` 并发回收
- COW触发时PTE的amoswap模拟是否正确

---

### Level 4: 较大内存占用
**目标**: 增加PTP分配数量，测试PTP区域的容量和回收能力
```bash
# 4a: 稍大的程序 — 更多的代码段映射
docker run --rm busybox find / -name "*.conf" 2>/dev/null | head -20

# 4b: 内存分配压力 — 触发更多的匿名页page fault
docker run --rm busybox sh -c "dd if=/dev/zero of=/tmp/testfile bs=1k count=512 && ls -la /tmp/testfile && rm /tmp/testfile"

# 4c: 大量小文件操作 — 频繁的mmap/munmap
docker run --rm busybox sh -c "for i in $(seq 1 50); do echo $i > /tmp/f_$i; done && ls /tmp/f_* | wc -l"
```
**关注点**:
- PTP区域的分配上限是否够用（检查OpenSBI的PTP池大小）
- 大批量PTP回收时的 `flush_reclaim_list` 是否会出现遗漏
- struct page* 的元数据（flags, refcount, lock）是否在批量场景下保持一致

---

### Level 5: 网络功能
**目标**: 测试网络namespace和veth设备相关的内核路径
```bash
# 5a: 基础网络（DNS解析 + HTTP请求）
docker run --rm busybox ping -c 3 127.0.0.1

# 5b: wget测试（如果网络通的话）
docker run --rm busybox wget -q -O - http://example.com 2>/dev/null | head -5

# 5c: 网络命名空间操作
docker run --rm busybox ip addr
```
**关注点**:
- 网络子系统中可能涉及的内存映射路径
- veth设备创建/销毁过程中的内核页表操作

---

### Level 6: 交互式容器（长生命周期）
**目标**: 测试长时间运行容器的稳定性，以及多次page fault的累积效果
```bash
# 6a: 交互式shell（手动退出）
docker run --rm -it busybox sh
# 进入后执行: ls, echo, cat /proc/self/maps, exit

# 6b: 限时后台运行
docker run --rm busybox sh -c "sleep 5 && echo done"

# 6c: 长时间运行 + 多次操作
docker run --rm busybox sh -c "for i in $(seq 1 20); do echo iteration_$i; sleep 0.5; done"
```
**关注点**:
- 长时间运行时Timer中断对NaCC路径的影响
- 多次进入/退出用户态的PTE权限切换是否正确
- `cat /proc/self/maps` 能帮助验证VMA布局是否合理

---

### Level 7: 多容器并发
**目标**: 测试多个容器同时运行时的资源隔离和PTP管理
```bash
# 7a: 两个容器顺序执行
docker run --rm busybox echo first && docker run --rm busybox echo second

# 7b: 两个容器并发执行（如果系统支持）
docker run --rm busybox sh -c "sleep 2 && echo A" &
docker run --rm busybox echo B
wait
```
**关注点**:
- 多个机密容器的PTP区域是否正确隔离
- 第二个容器是否能复用第一个容器回收的PTP空间
- 并发场景下OpenSBI的PTP分配/回收是否存在竞态

---

### Level 8: 更完整的应用
**目标**: 逐步接近真实应用场景
```bash
# 8a: Ubuntu基础镜像
docker run --rm ubuntu:22.04 echo "ubuntu works"

# 8b: Ubuntu基础操作
docker run --rm ubuntu:22.04 sh -c "apt list --installed 2>/dev/null | head -10"

# 8c: Python（如果有镜像且空间允许）
docker run --rm python:3-slim python3 -c "print('hello from python')"
```
**关注点**:
- 更大的ELF二进制带来的更复杂的VMA布局
- glibc动态链接器的mmap/mprotect操作路径
- 更深层级的页表结构（Level 0 PTE数量显著增加）

---

### 通用调试检查清单
每个测试完成后，确认以下几点：
1. **无 `BUG: Bad page state` 错误** — 表明struct page管理正确
2. **无 `BUG: Bad rss-counter` 错误** — 表明页面计数正确
3. **无 `non-zero pgtables_bytes` 错误** — 表明PTP回收完整
4. **页表摘要合理** — `pgtbl_debug` 输出中，退出后应只剩Agent的基础映射
5. **容器正常退出** — 无kernel panic, 无Oops, 无hang

### Lazy Coding 验证目标
NaCC的lazy coding核心理念是：**尽量少修改Linux内核，将页表保护的核心逻辑下沉到OpenSBI**。因此测试重点是验证：
- Linux的常规内存管理路径（`handle_page_fault`, `exit_mmap`, `copy_page_range`, `mprotect`）在NaCC修改后仍然语义正确
- OpenSBI的store/amoswap模拟覆盖了所有Linux会对PTP执行的写操作
- PTP分配/回收的生命周期管理在各种容器操作下都不泄漏/不越界

### 建议的测试顺序
> 按照 Level 2 → 3 → 4 的顺序逐步推进，每一个Level内部按 a → b → c 顺序测试。
> 前4个Level是最关键的，覆盖了绝大多数的内核内存管理路径。
> Level 5-8 是进一步的鲁棒性验证，可以在核心功能稳定后再进行。
