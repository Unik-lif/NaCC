## 2026-03-05 实验记录

### 背景回顾

昨日完成了 Phase 0：execve 路径修复（`NACC_INITED` 进程 → execve 场景）。今日目标是实现 Phase 1：fork + exec 场景的正确执行。

**昨日问题**：当 `NACC_INITED` 父进程执行 `fork()` 时，`dup_mmap()` → `copy_page_range()` 尝试遍历/读写父进程的页表页（PTP）。但 NaCC 已将 PTP 迁移至安全内存，普通内存访问触发 `anoand.d` 指令（安全内存专用指令），导致崩溃。

### 问题根因分析

```
fork() 调用链:
  kernel_clone()
    → copy_mm() → dup_mm() → dup_mmap()
      → 对每个 VMA 调用 copy_page_range(child_vma, parent_vma)
        → 遍历父进程 PGD → P4D → PUD → PMD → PTE
        → 读取 PTP 页内容（这些页在安全内存中！）
        → 触发 anoand.d → 崩溃
```

**核心矛盾**：`copy_page_range` 需要读取父进程的 PTP 内容来为子进程复制页表映射，但 NaCC 已将这些 PTP 页迁移到 M-Mode 才能访问的安全内存区域，S-Mode 的内核无法正常访问。

### 今日方案：弱化 fork（Phase 1: fork + exec 专用）

**设计思路**：fork + exec 场景下，子进程执行 exec 后会完全替换地址空间，因此 fork 时复制父进程地址空间是多余的。我们可以：

1. **在 fork 时跳过地址空间复制**：检测到父进程 `nacc_flag & NACC_INITED` 时，不复制页表映射
2. **子进程拿到一个"空白" mm**：有 pgd 但无用户态映射
3. **exec 时正常建立地址空间**：`load_elf_binary` 在新 mm 上按普通流程加载
4. **exec 完成后调用 `nacc_invoke_child()`**：重建 NaCC 保护

### 修改方案草案

```c
// linux/kernel/fork.c — dup_mmap()

#ifdef CONFIG_RISCV
#include <asm/nacc.h>
#endif

static __latent_entropy int dup_mmap(struct mm_struct *mm,
                                     struct mm_struct *oldmm)
{
    // ... 现有代码 ...

#ifdef CONFIG_RISCV
    /*
     * NaCC fork bypass: If the parent is a NaCC-protected process,
     * its PTP pages are in secure memory and cannot be accessed from
     * S-Mode. Skip the entire VMA/page-table copy.
     *
     * Phase 1 (fork+exec): The child will exec immediately, so it
     * doesn't need the parent's address space. The child gets an
     * empty user-space with only kernel mappings in the pgd.
     */
    if (current->thread.nacc_flag & NACC_INITED) {
        printk(KERN_ERR "[Linux]: dup_mmap: NaCC parent (pid %d), "
               "skipping address space copy for child\n", current->pid);
        // 清零子进程 mm 的统计信息
        mm->total_vm = 0;
        mm->data_vm = 0;
        mm->exec_vm = 0;
        mm->stack_vm = 0;
        mm->map_count = 0;
        // 跳过 VMA 遍历，直接到 out
        retval = 0;
        goto out;  // 或用合适的跳过方式
    }
#endif

    // ... 原有 VMA 遍历和 copy_page_range 逻辑 ...
}
```

### 风险与注意事项

1. **Phase 1 限定**：此方案仅适用于 fork + exec 场景。纯 fork（子进程不 exec）需要更复杂的处理（Phase 2）
2. **父进程恢复**：fork 过程中父进程的 mmap_write_lock 等锁需正常释放
3. **maple tree**：`__mt_dup()` 已在 VMA 遍历前调用，跳过策略时是否影响？需要评估是否也跳过 maple tree 复制
4. **子进程信号处理**：exec 前若子进程收到信号，空白 mm 可能导致问题（概率低但需注意）

### 对 Opus 4.6 计划的评审结论（Codex）

#### 结论
当前计划存在结构性风险，暂不建议按现稿实施。主要 blocker 是“fork 后给子进程空白 mm”这个前提不成立。

#### 主要问题（按严重级别）

1. `CRITICAL` 子进程空白 mm 无法安全走到 `execve`
- `fork` 返回后，子进程会先在原用户态执行，再发起 `execve`。
- `execve` 还需从用户态拷贝 `filename/argv/envp`。
- 若 mm 为空，可能在发起 `execve` 前崩溃，或在 `copy_from_user` 阶段失败。
- 结论：`fork+exec` 不能简单等价为“fork 时可丢弃全部用户映射”。

2. `CRITICAL` `dup_mmap()` 早退点设计不完整，可能破坏 mm/VMA 不变量
- 草案 `goto out` 与 `__mt_dup()`、VMA 遍历替换流程的关系未锁定。
- 在错误阶段早退可能造成 maple tree / VMA 状态不一致，风险为内核内存破坏级别。

3. `HIGH` 触发条件过宽
- 当前条件为“父进程 `NACC_INITED` 即跳过复制”。
- 会影响该进程所有 `fork`，不只“立刻 exec”的路径，行为回归面过大。

4. `HIGH` 手工清零 `mm` 统计字段不安全
- 仅清 `total_vm/data_vm/exec_vm/stack_vm/map_count` 过于局部。
- 易造成 `mm_struct` 相关状态不一致，不应作为主设计路径。

5. `MEDIUM` 测试覆盖不足
- 仅 `docker run busybox ...` 无法覆盖“child 在 exec 前执行用户态代码”的关键失败路径。
- 现有测试不能证明方案可行。

#### 建议
先重构技术前提，再确定 patch 落点；当前方案可保留为"问题分析草稿"，不应直接进入实现阶段。

### 修订方案：全权委托 OpenSBI 处理 fork 页表复制

经讨论和 Codex 评审，放弃"在 dup_mmap 内 hack"的思路。核心原因：**PTP 在安全内存中，只有 M-Mode 能访问**。应该让 OpenSBI 来做页表复制，与 NaCC 架构一致。

#### 设计概要

```
fork() 调用链（修改后）:
  kernel_clone()
    → copy_mm() → dup_mm() → dup_mmap()
      → VMA 元数据正常复制（maple tree, file mappings 等）
      → 检测 NACC_INITED → 跳过 copy_page_range()
      → 发起 SBI ecall: SBI_EXT_NACC_FORK(parent_pgd_pa, child_pgd_pa)
        → OpenSBI/SM: 遍历父进程页表（可访问安全内存）
        → 为子进程每级 PTP 分配新的安全内存页
        → 复制 PTE 内容到子进程页表
        → 返回成功
    → copy_thread(): 子进程 nacc_flag = NACC_FORKED
  → 子进程返回用户态，正常执行 execve
```

#### 新增 SBI ecall: `SBI_EXT_NACC_FORK`

**Linux 侧传递的信息**：
1. `parent_pgd_pa` — 父进程 PGD 物理地址（OpenSBI 据此遍历整个页表树）
2. `child_pgd_pa` — 子进程 PGD 物理地址（`pgd_alloc` 已在 `dup_mm` 中分配好）

**OpenSBI/SM 侧的工作**：
1. 遍历父进程页表树（M-Mode 可访问安全内存中的 PTP）
2. 为子进程的每一级 PTP 分配新的安全内存页
3. 将父进程 PTP 内容复制到子进程对应的 PTP 中
4. （Phase 1 暂不处理 COW，直接复制 PTE 原样）
5. 返回成功

#### 需要关注的风险

1. **子进程的安全内存 PTP 回收**：exec 或 exit 时需要回收 OpenSBI 分配的安全内存 PTP，需要对应的回收 ecall
2. **COW 延迟**：Phase 1 不做 COW 意味着父子共享可写页面，如果两者同时写同一页会数据损坏。fork+exec 场景下风险低（子进程很快 exec），纯 fork 场景必须解决
3. **TLB 一致性**：OpenSBI 修改父进程 PTE 后（如果做 COW），需要刷新父进程 TLB
4. **安全内存池容量**：fork 会使安全内存 PTP 页数量翻倍，需确认池是否够用
