## 实验记录
疑似是做了简易sys_clone的地址空间拷贝之后，出现了page fault没有处理过来的问题，先得检查一下

昨天看了一些rlcr的过程，觉得那个是适用于对东西非常熟悉的场景，但是我对linux本身还没有特别熟悉，还是按照vibe coding的方式多加点作为人的部分

和codex确认结果：
  1. 父进程为 NaCC 时，Linux 在 dup_mmap() 跳过 copy_page_range()，由 OpenSBI 的 nacc_fork(parent_pgd, child_pgd) 复制页表树。
  2. prototype 只保证 child 能继续运行到 execve()，不追求完整 Linux fork/COW 语义。
  3. OpenSBI 在复制页表时，对所有用户态可写 leaf PTE 统一做保守 wrprotect：
     - parent leaf PTE 清 W
     - child leaf PTE 也清 W
     - 不要求区分 anon/file/shared/private
  4. 目标是避免 child exec/exit 时误影响 parent，同时为 child exec 前的最小运行窗口提供足够的只读共享语义。
  5. 当前 Linux 侧已经有这些接点：
     - kernel/fork.c: NaCC parent 跳过 copy_page_range，并调用 nacc_fork(...)
     - arch/riscv/kernel/process.c: child 在 fork 后被标记为 NACC_FORKED
     - fs/exec.c: exec 时旧 mm 先走 NaCC reclaim，exec 完成后：
       - NACC_PREPARE -> nacc_invoke()
       - NACC_FORKED -> nacc_invoke_child()
  6. 需要在 OpenSBI/monitor 侧确认：
     - nacc_fork 复制每一级页表时，leaf 判定条件
     - 对 user writable leaf 的 wrprotect 规则
     - parent/child 页表页所有权与 reclaim 不互相污染
     - agent region / special mappings 是否跳过或特殊处理
