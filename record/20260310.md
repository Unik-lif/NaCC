# 2026-03-10 状态机梳理（NaCC fork/exec/re-exec）

## 背景

近期在 busybox `sh -c`、`fork+exec` 与同 PID `re-exec` 路径上出现了：

- Agent 二次初始化行为与预期不一致
- `Not in nacc process linux state. Simply omit it.` 卡住
- 回收路径偶发与预期不对齐

因此先暂停改功能，先统一状态机语义。

---

## 1. 当前状态定义（代码现状）

`thread.nacc_flag`（Linux）：

- `NACC_PREPARE = 0b001`
- `NACC_INITED = 0b010`
- `NACC_RECLAIM = 0b100`
- `NACC_FORKED = 0b1000`
- `NACC_REEXEC = 0b10000`

`CSR_NACC_STATE`（QEMU/OpenSBI 运行态）：

- `INACTIVE`
- `AGENT`
- `LINUX`
- `MONITOR`

---

## 2. Linux 侧状态机（现状）

```text
[NORMAL=0]
   | sys_nacc_register
   v
[PREPARE]
   | exec 成功后 bprm_execve 尾部 -> nacc_invoke()
   v
[INITED]
   | fork
   +-----------------------------> 子进程 [FORKED]
   |
   | same-pid re-exec:
   | begin_new_exec 前: |= RECLAIM
   | exec_mmap 后: = REEXEC
   v
[REEXEC]
   | bprm_execve 尾部 -> nacc_invoke_child()
   v
[INITED]

[INITED] --exit_group--> [INITED|RECLAIM] --exit_mmap/free_pgtables--> [DEAD]
```

说明：

- `INITED -> REEXEC -> INITED` 是同 PID re-exec 路径，用于避免再次 agent jump 初始化。
- `FORKED` 是 fork 出来的子进程标签，后续 exec 成功后走 `nacc_invoke_child()`。

---

## 3. 运行态状态机（CSR_NACC_STATE）

```text
INACTIVE --(nacc_invoke)--> AGENT --(acall)--> LINUX
LINUX --(aret on user trap)--> AGENT
LINUX --(switch to non-nacc task)--> INACTIVE
LINUX --(invoke_child/re-attach)--> LINUX
```

关键约束：

- `aret` 仅在 `nacc_state == LINUX` 时有效，否则 QEMU 会直接 omit（打印告警并 PC+4）。

---

## 4. 当前最可能的状态机缺口

### 缺口 A：`NACC_FORKED` 的 reclaim 前置不完整（高优先级）

现状里 `exec` 和 `exit_group` 主要按 `NACC_INITED` 去置 `NACC_RECLAIM`。  
但 `FORKED` 子进程若已经拥有安全 PTP（由 `nacc_fork` 构建），其 mm teardown 也应先进入 reclaim 语义。

风险：

- 子进程 exec/exit 时走普通释放路径，可能再次触发 bad page map/state 或隐式泄漏。

### 缺口 B：位标志与“等值判断”混用

`nacc_flag` 是位标志，但部分路径使用 `==` 判断。  
一旦引入组合态（如 `FORKED|RECLAIM`），某些分支会失配（例如 exec 尾部 re-attach 分支）。

### 缺口 C：Linux 标志与 CSR 运行态短窗不一致

虽然 `nacc_invoke_child()` 已把 `NACC_INITED` 延后到 SBI 调用后设置，但仍要保证：

- 任何触发 `aret` 的时刻，`CSR_NACC_STATE` 必须已经是 `LINUX`
- 失败路径不能把 Linux 留在“认为自己是 INITED，但 CSR 不是 LINUX”的状态

### 缺口 D：`FORKED` 子进程在 exec 前异常路径语义未完全封闭

包括但不限于：

- child 在 exec 前 `exit`
- child 在 exec 前再次 fork

这些路径是否允许、允许到什么程度，需要在状态机上显式定义。

---

## 5. 建议的收敛规则（先定规则再改代码）

先不纠结实现细节，先统一两条强约束：

1. 只要“当前 mm 持有安全 PTP”，在销毁 mm 前必须进入 `RECLAIM` 语义。
2. 只有在 re-attach 成功（SBI 完成、CSR 已是 `LINUX`）后，才能进入 `INITED`。

建议把状态语义抽象成：

```text
NORMAL
PREPARE
ACTIVE_INITED
ACTIVE_FORKED
TEARDOWN_RECLAIM
REATTACH_REEXEC
REATTACH_FORKED
DEAD
```

## 7. `config/vm_link.sh` 中 `AUTO_CMD` 测试映射

以下对应 `config/vm_link.sh` 里的现有命令模板，便于快速区分测试场景。

### 7.1 `docker run --security-opt seccomp=unconfined --rm busybox echo test`

场景类型：

- 直接执行 `echo`（无 `sh -c`）

主要链路（简化）：

```text
NORMAL -> PREPARE -> INITED -> (echo 执行) -> INITED|RECLAIM -> DEAD
```

用途：

- 验证最短路径是否稳定（注册、初始化、退出回收）

---

### 7.2 `docker run --security-opt seccomp=unconfined --rm busybox sh -c "echo hello > /tmp/test.txt && cat /tmp/test.txt"`

场景类型：

- `sh -c`，前段 builtin + 后段外部命令 `cat`
- 常见为同 PID re-exec 重点场景

主要链路（简化）：

```text
NORMAL -> PREPARE -> INITED (sh)
      -> RECLAIM -> REEXEC -> INITED (cat re-attach)
      -> INITED|RECLAIM -> DEAD
```

用途：

- 验证 re-exec 路径是否按 `REEXEC -> nacc_invoke_child()` 走
- 验证同 PID 切换后不会出现状态错位

---

### 7.3 `docker run --security-opt seccomp=unconfined --rm busybox sh -c 'echo hello; echo b; echo c'`

场景类型：

- 全 builtin（通常无外部进程、无 re-exec、无 fork）

主要链路（简化）：

```text
NORMAL -> PREPARE -> INITED (sh only)
      -> INITED|RECLAIM -> DEAD
```

用途：

- 作为对照组，验证“仅 shell 内建命令”时状态机与日志噪声

---

### 7.4 `docker run --security-opt seccomp=unconfined --rm busybox sh -c "cat /tmp/test.txt"`

场景类型：

- `sh -c` 单外部命令，busybox 常走同 PID re-exec

主要链路（简化）：

```text
NORMAL -> PREPARE -> INITED (sh)
      -> RECLAIM -> REEXEC -> INITED (cat re-attach)
      -> INITED|RECLAIM -> DEAD
```

用途：

- re-exec 最小复现命令（常用于复现/验证 Agent 二次初始化问题）

---

### 7.5 `docker run --security-opt seccomp=unconfined --rm busybox sh -c "cat /etc/hostname; echo done"`

场景类型：

- 典型 fork+exec：`cat` 后还有 `echo done`，父 shell 需继续执行

主要链路（简化）：

```text
父进程: NORMAL -> PREPARE -> INITED (sh parent)
子进程: (fork) -> FORKED -> (exec cat) -> [应 re-attach 到 INITED]
父进程: wait 子进程 -> 执行 builtin echo done -> INITED|RECLAIM -> DEAD
子进程: exec/exit teardown (应先满足 reclaim 语义) -> DEAD
```

用途：

- 验证 `FORKED` 路径是否完整
- 验证父子进程回收语义是否一致

## 9. 2026-03-10 16:05 `re-exec` 日志结论补充

对应日志：

- `logs/nacc_qemu_20260310_160520.log`
- `logs/nacc_vm_20260310_160520.log`

测试命令：

- `docker run --security-opt seccomp=unconfined --rm busybox sh -c "cat /tmp/test.txt"`

场景归类：

- 这是 **same-PID re-exec** 最小复现，不是 `fork+exec`

### 9.1 现象链

日志显示：

1. 首次 `nacc_invoke()` 成功，`sh` 已正常进入 NaCC。
2. `execve` 前正确置 `NACC_RECLAIM`，旧 mm 的 secure PTP 也确实走了 reclaim。
3. `exec_mmap` 后切到 `NACC_REEXEC`，随后走 `nacc_invoke_child()`。
4. OpenSBI 的 `sm_nacc_invoke_child()` 返回成功，并明确打印 `Child NACC_STATE set to ... (LINUX mode)`。
5. 紧接着出现 `BUG: Bad page state in process systemd-journal pfn:1bfff4`。
6. 随后 `cat` 在 `_new_vmalloc_restore_context_a0` 中 Oops，报
   `Unable to handle kernel access to user memory without uaccess routines`。
7. 之后才出现大量 `Not in nacc process linux state. Simply omit it.`，并最终 panic。

### 9.2 当前判断：`Bad page state` 不是直接死因

`Bad page state` 说明 Linux 侧某个 NaCC 页表页的 `struct page` / `ptdesc` / ptlock metadata
有问题，但它 **不足以单独解释** 后面 `cat` 立刻以“kernel trap 访问用户地址”的方式崩溃。

更直接的致命现场是：

- Oops 点落在 Linux trap 入口 `_new_vmalloc_restore_context_a0`
- 当时寄存器里的 `tp` 是用户地址
- `badaddr = tp + TASK_TI_A0 offset`

这说明 Linux trap 入口把 `tp` 当成 kernel thread pointer / `thread_info` 使用了，
但它实际仍像是用户态 TLS 指针。于是 trap entry 代码直接去访问用户地址，导致 Oops。

因此，`cat` 的直接死因更像是：

- **agent / monitor / Linux 三者之间的 trap 上下文恢复错位**
- 尤其是 `tp` / `CSR_SSCRATCH` / host trap context 没有在 re-exec 后被正确刷新

### 9.3 更关键的设计结论：same-PID `re-exec` 不能复用 `fork+exec` 的 `invoke_child` 流程

当前 `NACC_REEXEC -> nacc_invoke_child()` 的做法，实质上把
“same-PID re-exec” 当成了“fork child re-attach”。

这条假设现在看是错误的。

原因：

- `fork child` 场景中，child 是一个新任务，复用 parent 的 agent 语义是合理的；
- same-PID `re-exec` 场景中，PID 没变，但 **用户态映像、exec 建立的新 mm、trap 上下文前提都变了**；
- `nacc_invoke_child()` 当前只做了：
  - child/re-attach 注册
  - PTP transfer
  - agent region 映射
  - `CSR_NACC_STATE = LINUX`
- 但它 **没有重新建立首次 `nacc_invoke()` 才会准备的 agent/host trap 协作上下文**，
  包括 `user_pt_regs`、`do_irq`、`excp_vect_table`、`current_gp` 这类入口参数。

因此，same-PID `re-exec` 若继续复用 `fork+exec` 的 `invoke_child` 路径，
很容易出现：

- 页表似乎 re-attach 成功
- `NACC_STATE` 也已经切到 `LINUX`
- 但第一次真正用户 trap 回 Linux 时，上下文仍沿用旧进程映像的假设
- 最终在 trap 入口处炸掉

### 9.4 与 `Bad page state` 并存的另一条问题线

本次日志里，`pfn 1bfff4` 并不是崩溃后才出现的异常页。

它在 `sys_execve` 期间、`NACC_RECLAIM` 置位之前，就已经作为 NaCC PTE page 被 Linux 分配了。
随后 `sm_nacc_invoke_child()` 又对新的 mm 做了一次 `sm_transfer_ptp()`。

日志中甚至出现了：

- `old_pfn->new_pfn: 1bfff5 -> 1bfffe`

这说明：

- re-exec 的新 mm 在 Linux 侧已经先创建了一批 NaCC secure PTP
- OpenSBI 又把这批页当成“old page table”再次做 secure transfer

这会把以下语义搅乱：

- `nacc_mappings`
- `pagetable_pte_ctor/pmd_ctor`
- ptlock
- reclaim / free 路径

所以这次故障更像是 **两条问题线同时暴露**：

1. re-exec trap context 刷新语义不完整
2. re-exec 新 mm 的页表页 metadata / transfer 语义也不闭合

但从“谁先打死 `cat`”来看，第一条更像首因。

### 9.5 当前建议

短期设计建议：

- **不要再让 same-PID `re-exec` 复用 `fork+exec` 的 `nacc_invoke_child()` 流程**
- `NACC_REEXEC` 应该有单独语义，至少与 `NACC_FORKED` 分流

建议单独定义一条 re-exec 路径，例如：

- 方案 A：真正重新走一遍轻量版 `nacc_invoke()`，显式刷新 agent/host trap context
- 方案 B：保留“不跳 agent 初始化”的优化目标，但必须补一个专门的 re-exec context refresh ecall

在没有把 re-exec trap context 语义理顺前，不应再假设：

- `NACC_REEXEC` 只是 `NACC_FORKED` 的“同 PID 版本”

### 9.6 后续验证点

- 比较首次 `sh` invoke 与 `cat` re-exec 后，agent 中保存的
  `_user_pt_regs` / `_do_irq` / `_excp_vect_table` / `_current_gp` 是否仍是旧值。
- 在 `cat` 第一次用户 trap 前后抓取 `tp` / `CSR_SSCRATCH` / `sepc` / `scause`，
  验证是否返回 Linux trap 时 `tp` 仍是用户 TLS。
- 单独区分：
  - “trap context 刷新问题”
  - “re-exec 新 mm 的 NaCC PTP metadata/transfer 闭环问题”

当前优先级判断：

- **优先修 re-exec 不能复用 fork-child 流程这个设计错误**
- `Bad page state` 问题随后继续收敛
