## 实验记录
这几天主要卡在fork的完整标准实现，以及一些边角问题上

SEGV_ACCERR：地址落在合法映射里，但是访问类型似乎不被允许

### 2026-03-24 到 2026-03-25 这轮调试的非正式总结

这轮先修掉了一个真实 bug，但它不是最终问题。

1. child attach 的 pid 语义不一致
- parent early register 原先用的是 `pid_vnr()`，child attach / OpenSBI 查表用的是 global pid
- 结果就是 register 记成 namespace pid，attach 用 global pid 查表 miss
- 修法是把 early register 改成 global pid，并且 attach 失败时对 child fail-fast，避免卡在 `NACC_FORKED` 反复重试
- 这部分修完之后，`child pid ... is not registered` 确实消失了

2. 但 fork+exec 还是没有真正通过
- `docker run --security-opt seccomp=unconfined --rm busybox sh -c "cat /etc/hostname; echo done"`
  还是经常没有 `hostname` / `done`
- 所以 attach 修好只是把坏点往后推，不是最终修复

3. 后面做了大量排错，逐步排除了一些很像的假根因
- `filemap_map_pages` 本身不是“完全没装进去”
- `do_wp_page()` 是正常 Linux 路径，不是 NaCC 特有异常
- `wp_page_copy()` 的主干事务目前看是通的：旧 file-backed PTE 清掉后，新 anon writable PTE 能装回去
- `d3` / `93` 这条位变化，后来发现很大一部分只是 parent / child 在不同 mm 里的正常差异，不是单个 PTE 自己在抖

4. OpenSBI 这边也做了一个机制收敛
- secure `ptep_get_and_clear()` 不再靠 trap 里对 `AMOSWAP` 的隐式模拟
- 现在改成显式 `GET_AND_CLEAR_PTE` SBI 路径
- 这个改动提高了可观测性，也让 Linux / OpenSBI 两边的旧值、新值更容易对起来
- 但它也没有直接修掉最终问题，只是排除了“clear 机制本身不对”这个怀疑

5. 这轮最大的认知修正其实是对 `SEGV_ACCERR`
- 一开始很容易被日志里那句 `code is set to SEGV_ACCERR` 误导
- 但后来真正把 `fault.c` 的 `fault_pte` 打出来后，发现很多 secure mm fault 实际上：
  - fault 前 `pte=0`
  - `access_error=0`
  - `handle_mm_fault()` 后 `pte` 就变成有效 leaf PTE
- 也就是说，这里面有不少只是正常 not-present fault，不是“权限位明确错了”

6. 当前更像的问题
- 不是单次补页完全坏掉
- 更像是：很多局部 fault 都能补好，但整体执行没有收敛到用户可见输出
- 换句话说，现在更像“进展问题 / fault storm / 执行路径没有跑透”，而不是单个 PTE 写错

7. 这轮经验值得记住
- 不要一看到 `SEGV_ACCERR` 就先入为主地归因给权限位
- 不要把同一个 VA 在 parent / child 两个 mm 中的 PTE 混为一谈
- `do_wp_page/wp_page_copy` 和 `mprotect` 这些路径很多时候是正常动态链接器行为，不能看到 clear 就先当 bug
- 现在更应该补的是“进展观测”，比如：
  - 是否真的走到 `write(1, ...)`
  - 同一地址是否在异常重复 fault
  - 用户态 PC 是否在推进
