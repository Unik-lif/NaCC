## 2026-03-30

### Informal Progress Summary For The Recent Round

These days the project made real progress, but the main user-visible target is still not finished.

### 1. `fork + exec` is still not functionally complete

- The key negative fact has not changed: the user terminal still does not reliably show expected output for the important `fork + exec` smoke cases.
- In particular, commands such as:
  - `docker run --security-opt seccomp=unconfined --rm busybox sh -c "cat /etc/hostname; echo done"`
  still cannot be treated as passed.
- So even though several internal paths have improved, we still cannot claim that `fork + exec` is working end-to-end.

### 2. Recent code movement did improve the runtime model

Recent subrepo history shows that the current round was not only about old fork cleanup; it also started moving toward multi-process runtime virtualization:

- `linux`:
  - `4d7101e02c1b` `[CODE]: refresh fork child trap contract via agent attach`
- `opensbi`:
  - `821c800` `[CODE]: add per-pid NaCC thread context handoff`
- `qemu`:
  - `7c543608af` `[CODE]: allow OpenSBI NaCC CSR writes from M-mode`

This is important because it means the work is no longer purely stuck on one old `fork` hypothesis. The system has started to gain the basic machinery needed to distinguish multiple NACC processes at runtime.

### 3. Multi NACC process progress: two processes were successfully observed

- A meaningful intermediate result of this round is that we were able to observe two NACC processes instead of remaining trapped in a purely single-process runtime model.
- This is still buggy and incomplete, but it is an important direction change:
  - the project is no longer only trying to "repair one fork path"
  - it is also beginning to virtualize runtime state for multiple protected processes
- This should be treated as partial progress, not as proof that multi-process support is already stable.

### 4. New problem exposed: Heisenbug-shaped observability failure

While trying to debug the remaining issues more closely, a new and serious problem became clear:

- adding or changing `printk` / trace points in hot paths such as:
  - `do_trap_ecall_u`
  - signal-related paths
  - other trap / return hot spots
  can materially change the behavior of the whole program
- this is not just "logs are noisy"; it looks more like a Heisenbug-shaped runtime sensitivity problem

The clearest immediate evidence is that:

- `logs/nacc_qemu_20260330_105927.log`
- `logs/nacc_qemu_20260330_153016.log`

stop at very different points, even though they belong to the same investigation direction.

That means the system is now sensitive enough that extra observability itself can perturb control flow in a major way.

### 5. Why this matters

This problem matters for two reasons:

1. It weakens debugging confidence.
- If hot-path logging can move the stop point significantly, then a lot of "evidence" from verbose tracing becomes hard to trust as a stable description of the real bug.

2. It is also a future performance concern.
- The project will eventually need to evaluate NaCC performance.
- If the current runtime behavior is highly sensitive to instrumentation, then we should not keep expanding heavyweight trace-based debugging without a lower-perturbation observability strategy.

### 6. Current interpretation of the round

The current state is best summarized as:

- `fork + exec` still does not pass in a user-visible way
- multi-NACC-process runtime work has started and already exposed two protected processes
- but the system now also shows Heisenbug-shaped behavior under added logging

So this round was not a failure, but it also was not a semantic pass:

- we gained runtime-model progress
- we exposed a new debugging/performance risk
- and we still do not have a stable end-to-end `fork + exec` result

### 7. Practical takeaway

The most important practical takeaway from this round is:

- from now on, heavy hot-path logging should be treated with caution
- differences caused by added trace points must not be casually interpreted as stable root-cause evidence
- the project likely needs a lower-perturbation observability style for the next debugging stage
