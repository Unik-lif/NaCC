## 2026-04-07

### Informal Progress Summary For The Recent Round

This round marks a real architectural milestone rather than another local fork fix:

- `fork + exec` now works on the native path
- the project no longer depends on the old bulk-style fork model
- the project no longer depends on `transfer_ptp` for the active `exec` path

The meaning of this progress is important: NaCC is no longer only "trying to repair a prototype fork path". It now has a much stronger claim to running on the intended Linux-native page-table construction path.

### 1. `fork + exec` is now considered functionally complete for the current milestone

- The previous blocker was that `fork + exec` still did not reliably produce visible user output.
- That is no longer the accepted state of the project for this round.
- The current milestone is that `fork + exec` has been finished without falling back to the old bulk fork path or the old `transfer_ptp`-based exec rebuild.

This does **not** mean every later container semantic is already validated. It means the core `fork + exec` milestone has crossed from "still blocked" into "working on the intended direction".

### 2. The active model is now closer to direct secure page-table ownership

The recent code movement indicates that the project has shifted away from the older "build normally, then bulk replace/transfer" style:

- `linux`:
  - `d8cf8d4bb0d0` `[CODE]: direct secure ptp exec path and remove remap leftovers`
- `opensbi`:
  - `f5458bb` `[CODE]: validate native secure ptp exec path and drop legacy marks`

The architectural meaning is:

- fresh exec address-space construction is now much closer to direct secure PTP allocation
- the old PFN-remap / remap-leftover model has been reduced
- the old `transfer_ptp` bootstrap is no longer the active answer for the main `exec` path

This is a stronger and cleaner direction than the earlier transitional model.

### 3. Bulk-style fork is no longer the mainline semantic path

- The project had already been moving away from the legacy bulk `nacc_fork` / `sm_nacc_fork` model.
- This round makes that direction substantially more concrete:
  - `fork + exec` no longer needs to be justified by a prototype bulk-copy escape hatch
  - the active path stays closer to native Linux `dup_mmap()/copy_page_range()` and direct secure PTP handling

So the project should now treat bulk-style fork as historical compatibility logic rather than the path to optimize around.

### 4. The old `transfer_ptp` model should now be treated as transitional / legacy

Historically, `transfer_ptp` was doing too much:

- converting an already-built Linux user page table into secure ownership
- carrying old/new PFN remap assumptions
- supporting a more bulk-style bootstrap model

That model was useful during bring-up, but it also made the system harder to reason about and harder to optimize.

The new progress means:

- `transfer_ptp` is no longer the architectural center of the active `exec` path
- direct secure PTP construction is now the preferred direction
- remaining dependencies on old transfer/remap machinery should be treated as cleanup or compatibility debt unless proven otherwise

### 5. Complexity has been reduced in a meaningful way

A major benefit of this round is not only that one more testcase passes. The model itself has become simpler:

- fewer "replace later" assumptions
- less dependence on PFN remap indirection
- fewer reasons to preserve bulk migration logic as first-class architecture
- a cleaner story for future performance work

This simplification matters because many earlier debugging and correctness problems were entangled with transitional machinery, not only with the final intended design.

### 6. What this round does **not** prove yet

Even with this milestone, several larger semantic claims are still not automatically proven:

- multi-child / `wait` behavior
- pipe / pipeline behavior
- shared-memory / `mmap` / `MAP_SHARED`
- broader multi-process stability under repetition
- final runtime-context hardening for multiple protected tasks

So this round should be treated as:

- a real architectural success
- but not yet the end of container-semantic validation

### 7. Updated interpretation of project state

The project state is now better summarized as:

- `fork + exec` has crossed the basic milestone on the intended native path
- direct secure PTP construction has replaced a large part of the earlier transfer/remap model
- the next phase should shift from "can `fork + exec` pass?" toward "how broad and stable are the container semantics now?"

That is a healthier place to plan from than the earlier "still blocked on visible output" state.

### 8. Immediate planning consequence

The next step should no longer be another large redesign of fork/exec bootstrap.

The next step should be a **coverage-first validation round** focused on the semantics that naturally come after `fork + exec`:

- pipelines
- `wait` / multi-child behavior
- shared mappings
- repeated runs and multi-process stability

If those fail, the next debugging target should be chosen from concrete semantic gaps rather than from the old bulk-transfer architecture.
