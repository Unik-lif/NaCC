## 2026-05-12 22:43:19 +0800 - leaf data lifecycle 日志理解

这次 ordered workload 1..8 的现象可以分成两个层面：

1. 用户可见功能路径跑通了。launcher 都是 `result=ok`，VM 侧命令输出符合预期，SSH auto exit 都是 `code=0`。
2. 内核日志不干净。每个 workload 的 QEMU 日志都有 `BUG: Bad rss-counter state`，第一处坏点在 workload 1 的 QEMU log line 8099，`MM_FILEPAGES val:143`，紧跟 `MM_ANONPAGES val:29`。

我的理解是：当前实现把已经存在的 user leaf PTE 转成了 `pte_special()`，并给 VMA 设置了 `VM_MIXEDMAP`，这样后续 Linux 普通路径里的 `vm_normal_page()` 不再把这些 PTE 当成普通 `struct page` 映射。这符合“从普通 Linux 生命周期中脱离”的方向，但还没有把 Linux 已经建立起来的 RSS/rmap/LRU/refcount 等账同步剪掉。

证据很强的一点是 workload 1 中：

- `nacc_detach_user_leaf_pages tag=nacc_attach_forked_child ... detached_ptes=172`
- 后面 RSS BUG 为 `MM_FILEPAGES val:143` + `MM_ANONPAGES val:29`
- 两者相加正好是 `172`

这说明问题不是 workload 本身崩掉，而是 detach 之后，Linux teardown/zap 看到 special PTE 时走了 generic special mapping 分支，只清 PTE，没有走普通 folio 分支，因此没有做 RSS decrement / rmap removal / folio lifecycle 处理。这个 bug 应该可以修：在正常 Linux zap/teardown 路径里识别 NaCC special private leaf，走 NaCC 专用 cleanup 分支，补上 Linux 侧 accounting，再必要时通过 SBI ecall 通知 Monitor 清 private bitmap / metadata。

需要区分的是：SBI ecall 不能替 Linux 直接处理 RSS/rmap，因为这些是 Linux 内部数据结构；比较合理的是 Linux 负责普通页生命周期账，SBI/OpenSBI 负责 private bitmap、root_l0 record、secure PTP 等 monitor-owned metadata。

teardown 当前“看起来能跑完”的原因是：

- secure PTP 页表页回收路径基本通了：Linux free page table 时识别 NaCC secure PTP PFN，加入 reclaim list，再通过 `sm_reclaim_ptp()` 让 OpenSBI `sec_mem_free()`。
- root_l0 retire 路径也通了：PGD/root free 时 `nacc_retire_root_sbi()` 触发 OpenSBI `nacc_bitmap_retire_root_l0()`，日志中有 `bitmap: retire root_l0 root_pfn=...`。
- process unregister / thread_ctx remove 也执行了。

但这不等于 leaf data 的普通页生命周期正确结束。当前缺的是 private leaf data 对应的 Linux accounting cleanup。

另一个更重要的功能问题是 post-attach page fault。日志里 page fault 新增的页表页 PTP 是 secure mem 分配的，但新 fault 出来的 leaf data page 看起来仍然是普通 Linux page。并且这次 patch 移除了 `SET_PTES` / `UPDATE_PTE` 阶段的 eager private bitmap tagging，所以 attach 之后新出现的 leaf 不会自动进入 private bitmap，除非后续再有一次 sync/tag。

COW 应该被视为 page fault 大类中的一个重点子情况：fork 后写共享页会走 write-protect fault，进入 `do_wp_page()` / `wp_page_copy()`。但由于当前 PTE 被做成 special，`vm_normal_page()` 可能返回 NULL，Linux 可能走 PFN/special mapping 风格路径，COW 后新页也可能变成普通匿名页并重新建立普通 RSS/rmap。因此后续处理 page fault 时必须覆盖 COW fault，但 fork/copy_page_range、mprotect、mremap、zap/exit/munmap 等不一定都能靠 fault handler 兜住，需要分别确认。

给 planner 的建议表达：

- 当前路线不是失败，workload 功能层面通过，但验证结果应标为 suspicious。
- RSS BUG 是可修的 lifecycle/accounting 漏洞，优先修 Linux zap/teardown 对 NaCC special private leaf 的 cleanup。
- post-attach page fault 是下一阶段核心功能缺口，必须定义 fault 新增 leaf 如何进入 private bitmap，以及 COW fault 是否同步完成 detach/tag/accounting。
- 下一步可以开一个 coder repair packet，先做最小闭环：NaCC special private leaf 在 zap/exit 路径里正确扣 RSS/rmap，并保留或补充 SBI metadata cleanup；随后单独规划 post-attach fault/COW 处理。

补充强调：teardown 路径里可能必须显式处理 NaCC process 的 special user leaf，而不能只依赖通用 special mapping 分支。当前日志能证明 secure PTP 页表页已经通过 reclaim list / `sm_reclaim_ptp()` 被回收，root_l0 也通过 `nacc_bitmap_retire_root_l0()` 退场；但这不能证明 NaCC process 的用户数据页已经正确释放。由于这些 leaf PTE 已被标成 `pte_special()`，普通 `zap_present_ptes()` 里的 `vm_normal_page()` 可能返回 NULL，进而跳过普通 folio 的 RSS/rmap/LRU/refcount/free 路径。因此后续修复不应只理解成“扣 RSS counter”，而应理解成“对 NaCC special private leaf 做完整 teardown”：Linux 侧找回对应 PFN/folio 并释放普通页生命周期账，OpenSBI/SBI 侧清理 private bitmap / ownership metadata。否则很可能只是清掉了 secure PTP 和 root record，但用户数据 leaf page 本身没有真正从 NaCC/private 状态和 Linux page 生命周期中正确退场。
