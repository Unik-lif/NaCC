# 2026-05-13 NaCC leaf data lifecycle 下一步计划

## 当前判断

今天这轮测试的总体感觉是：workload 大体可以跑到用户可见输出，但回收路径还没有稳定。

前一轮修复已经把最初那个“大块 detached leaf 全部没有走 Linux folio cleanup”的问题明显推进了一步。尤其 workload 5 之前出现过 `MM_FILEPAGES 153 + MM_ANONPAGES 30 = 183`，刚好等于 `detached_ptes=183`；这轮没有再看到这种完整 detached-leaf-sized residual。

但 T1 仍然不能关闭。workload 3 到 8 仍然出现 `BUG: Bad rss-counter state`。现在的主要形态变成：

- 小规模 `MM_ANONPAGES` 残留，例如 workload 3 的 `MM_ANONPAGES val:8`；
- 父进程/container shell teardown 附近反复出现 `MM_FILEPAGES val:-1` 加 `MM_ANONPAGES val:3`；
- workload 4 和 6 在输出 `done` 之后 SSH timeout，并且 QEMU 日志后续进入 RCU/workqueue/watchdog soft lockup；
- workload 5 和 7 出现 exit `139`，但这轮日志没有复现之前的 `epc=0` / `badaddr=0`。

这些现象说明当前问题已经不是单纯的“所有 NaCC special private PTE 都被 generic special zap 跳过”。更像是 fork/exec/post-attach/page fault/COW 过程中产生或替换的小批 leaf page，没有和 Linux RSS/rmap/refcount teardown、OpenSBI private-data tag 生命周期形成守恒闭环。

## 重要边界

即便 post-attach page fault / COW 的完整安全策略还没有完成，Linux mm 回收时也不应该出现 RSS counter 不守恒。

因此下一步不应该直接跳到“大设计：所有 page fault 新页如何纳入 OpenSBI 生命周期”。更稳的路线是先把回收路径做成可解释、可守恒：

1. Linux 侧 RSS/rmap/refcount/LRU-style cleanup 必须在 mm exit/munmap/exec/zap 时正确完成；
2. OpenSBI private bitmap / root_l0 / secure PTP metadata cleanup 不能替代 Linux accounting cleanup；
3. 如果某些 page fault/COW 新页尚未纳入 NaCC private bitmap，也必须在 teardown 日志里能明确说明它们为什么残留、在哪里加账、在哪里没有减账。

## 下一步推荐路线

建议开一个 bounded coder third pass，目标只聚焦 RSS teardown/accounting 收敛，不扩展完整 page fault/COW policy。

### 目标 1：解释并修复 workload 3 的 `MM_ANONPAGES val:8`

workload 3 的命令大致是：

```sh
a=seed; (a=long-string; :); echo fork_private_done
```

括号会产生 subshell/fork child，child 中修改变量会触发匿名页写入，可能涉及 fork 后 COW 或 post-attach page fault。

关键现象：

- child attach 日志里有 `nacc_attach_forked_child ... detached_ptes=170 already_special=3`；
- 退出时出现 `MM_ANONPAGES val:8`；
- `8` 远小于 `170`，所以不像最初整批 detached PTE cleanup 被跳过，而像是 fork/COW/page fault 后新增或替换的一小批匿名页没有被 teardown 正确覆盖。

需要 coder 追踪这 8 页：

- 它们在哪里被 `inc_mm_counter(mm, MM_ANONPAGES)` 或等价路径加账；
- teardown 时对应 PTE 是 normal、special、migration、none 还是已经被 NaCC 改写；
- 是否命中了 NaCC special private zap 分支；
- 如果没有命中，识别谓词为什么拒绝；
- 如果命中，`zap_present_folio_ptes()` 是否按 anon 类型减账。

### 目标 2：解释 `MM_FILEPAGES val:-1` + `MM_ANONPAGES val:3`

这个组合比单纯残留更危险，因为它显示出类型不匹配或过度 cleanup 的可能：

- `MM_FILEPAGES val:-1` 表示 file RSS 被多减了 1 页；
- `MM_ANONPAGES val:3` 表示 anon RSS 仍然少减了 3 页；
- 二者同时出现，说明可能存在某个路径把应当按 anon 处理的页按 file 类型减掉，或者同一个小集合里的页面经历了重复 cleanup / 错误分类。

需要重点检查：

- NaCC special private zap 分支进入 `zap_present_folio_ptes()` 时，`folio_test_anon(folio)` 和 `mm_counter(folio)` 的结果；
- PTE 原本的 VMA/RSS 类型和 folio 当前类型是否一致；
- fork/exec 后父 shell/container shell 的 mm 是否存在已经特殊化的旧 PTE 和新 fault 出来的普通 PTE 混合；
- `retire_ret == 0` 的 tag-already-clear 情况下，是否有重复处理同一 PFN/folio 的可能。

### 目标 3：补最小化诊断，而不是直接猜修

建议先加临时或 ratelimited 诊断，围绕 NaCC teardown 分支记录：

- `pid/comm/mm/addr/pfn/vma flags/pte flags`；
- 是否 active NaCC mm、是否 protected user-half、是否 `pte_special()`、是否 `VM_MIXEDMAP`；
- `retire_ret` 和 `retired`；
- `folio_test_anon(folio)`、`mm_counter(folio)`；
- 进入 cleanup 前后的 RSS delta；
- 对 NaCC mm 中没有命中特殊分支的 special PTE 做计数，而不是只静默走 generic special clear。

目标是让下一轮日志能够回答：RSS BUG 是漏减、错减，还是重复减。

## Page fault / COW 后续任务

page fault 和 COW 仍然是下一阶段必须解决的问题，但建议单独开包。

后续设计问题是：NaCC-active mm 在 VPN2 `0..255` 范围内通过 fault/COW/mmap/brk/stack growth 产生或替换用户 leaf 时，Linux 和 OpenSBI 怎么协作：

- 新 leaf 如何进入 private-data bitmap；
- 旧 leaf 被替换时如何完成 Linux accounting/rmap/refcount cleanup；
- COW replacement page 如何继承或重新建立 NaCC private state；
- fork child inherited mapping 是否需要 per-mm/per-mapping 状态，而不是只依赖 PFN-wide private-data tag；
- 不能回到 VMA/file/anon/heap/stack/code 语义分类作为安全边界。

但在 RSS counter 仍然不守恒时，不建议把这个大设计和当前 teardown 修复混在一个 patch 里。

## 推荐执行顺序

1. 先做 bounded teardown/accounting third pass，目标是让 workload 3/4/5/7/8 的 RSS BUG 消失，或至少每个 residual 都能精确归因。
2. 如果第三 pass 证明残留来自 post-attach fault/COW 新页，则只补 teardown 侧的守恒修复，不顺手扩展完整 security policy。
3. 再开独立 page fault/COW lifecycle packet，定义新页进入 OpenSBI private-data 管理的机制。
4. 最后重新跑 ordered workloads `1..8`，区分用户可见结果、RSS accounting、tag-already-clear 诊断、exit `139` 和 soft-lockup 这几类信号。

一句话结论：先保证回收时 Linux 账本不坏，再系统性处理 post-attach page fault/COW 的安全生命周期。
