## Important correction on COW retire semantics

The COW-side `retire_private_pfn` call is not a complete lifecycle design.

The earlier local reasoning, "COW replaces this old PTE, therefore retire the old PFN tag", is only safe if the PFN tag belongs uniquely to that mapping, or if the trusted side has owner/refcount state. Current OpenSBI `retire_private_pfn` clears the PFN-wide bitmap tag directly:

```text
PRIVATE_DATA -> NORMAL
```

It does not check whether a parent process or another fork child still maps the same PFN. Therefore, in a fork/COW scenario:

```text
parent PTE -> PFN X, tag=PRIVATE_DATA
child  PTE -> PFN X, tag=PRIVATE_DATA

child COW:
    child PTE -> new anon page
    parent PTE still -> PFN X
```

clearing PFN X's tag during the child COW path could be semantically wrong. The parent may still hold a confidential mapping to PFN X. This is why the residual `tag already clear` diagnostics should be treated as a separate tag lifecycle risk, not as proof of correctness.

## Security boundary for refcount ownership

Linux-side refcounting is acceptable for Linux accounting, but it must not become the authority for `PRIVATE_DATA -> NORMAL`.

Linux may report lifecycle events:

- fork-copy inherited a NaCC private PFN;
- COW replaced an old NaCC private PFN in one mapping;
- zap/munmap/exit removed a mapping;
- exec teardown removed old mappings.

But Linux should not be able to decide that a PFN can be declassified. Otherwise a malicious or compromised Linux could fake a final release and open access to a still-live NaCC process page.

The durable security rule should be:

```text
OpenSBI/Monitor owns the bitmap tag transition.
Linux is only an untrusted event source.
Trusted metadata decides whether a PFN still has a live NaCC owner.
```

## Agent-resident trusted metadata direction

A promising direction is to keep hard tags in OpenSBI while moving larger lifecycle metadata into protected Agent memory.

Proposed split:

```text
OpenSBI / Monitor:
    keep bitmap tag authority;
    validate caller/CID/root/PFN event context;
    update PRIVATE_DATA/NORMAL tags;
    keep code and static state small.

Agent-resident trusted metadata:
    store larger PFN lifecycle state;
    track owner/ref/shared state for private data PFNs;
    be inaccessible to Linux;
    be usable by future Agent management logic.

Linux:
    report lifecycle events only;
    keep ordinary RSS/rmap/refcount accounting;
    never directly clear private-data tags.
```

This is not a trust downgrade if the Agent metadata region is protected. It is a storage/layout move inside the TCB: large owner/ref tables live in Agent-resident protected memory instead of expanding the OpenSBI image or core static region.

The preferred implementation shape is not an expensive per-event full Agent entry:

```text
Linux -> OpenSBI -> enter Agent -> return OpenSBI -> Linux
```

Instead, OpenSBI can directly read/write a protected Agent metadata arena by physical address:

```text
Linux -> ecall -> OpenSBI
OpenSBI validates event
OpenSBI updates Agent-resident metadata directly
OpenSBI clears bitmap only when trusted metadata says last owner is gone
```

Agent can map the same physical metadata pages into its own virtual address space for future management/compaction/diagnostics, but OpenSBI should use physical addresses to avoid depending on Agent virtual translation.

## Space and fast-path policy

The bitmap should remain the fast hardware-like authority for access classification. Larger owner/ref metadata should not be consulted on every load/store.

Possible state split:

```text
NORMAL
PRIVATE_UNSHARED
PRIVATE_SHARED
```

The cheap path:

- unshared private page release can be handled directly by OpenSBI;
- fork-copy or other sharing upgrades a PFN to tracked/shared state;
- only shared/ambiguous lifecycle events consult the Agent-resident metadata.

This keeps ordinary access and most simple lifecycle cases cheap, while avoiding unsafe PFN-wide tag clearing for fork/COW.

## Open follow-up

Open a separate planning/design packet before changing this path further. The next packet should answer:

- exact metadata location: Agent region carve-out vs existing metadata region;
- metadata protection from Linux and ordinary user code;
- minimal record key: PFN only, `{cid,pfn}`, or `{root,cid,pfn}`;
- fork-copy increment semantics;
- COW/zap/exit decrement semantics;
- batch ABI for teardown paths;
- when OpenSBI may clear `PRIVATE_DATA` to `NORMAL`;
- how to avoid per-PTE ecall overhead in large zap/exit paths.

Additional follow-up: current COW replacement still installs an ordinary Linux anon page. That was acceptable for the RSS-accounting repair, because the packet only needed Linux's old/new folio accounting to be conserved. It is not the final NaCC private-data lifecycle policy. A future COW lifecycle design must decide how the replacement page re-enters NaCC private ownership, whether through OpenSBI tagging, Agent-resident lifecycle metadata, or another trusted handoff.

Do not treat the current RSS repair as the final tag lifecycle policy.
